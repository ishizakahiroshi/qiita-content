先に結論から書きます。

`echo "$password" | sudo -S tee /path/to/file` のように、**パスワードとファイルの中身を同じ標準入力に流し込む書き方はやめたほうがいいです**。sudo の認証がキャッシュされている状態だと、`sudo -S` が 1 行目のパスワードを読み取らず、そのパスワードが後ろの中身と一緒にファイルへ書き込まれることがあります。

しかもそのファイルが Web で配信される場所に置かれていると、パスワードが外から取得できる状態になります。実際にやりました。やってしまった側の話として書きます。

![記事全体の要約（漏洩のメカニズムと経路分離の対策）](https://raw.githubusercontent.com/ishizakahiroshi/qiita-content/main/public/images/2026-07-07_sudo-s-stdin-password-leak_infographic.png)

## 自動化に任せていたら、公開ファイルにパスワードが書かれていた

ある Web アプリの運用をしていて、サーバー上の静的ファイルを 1 つだけ差し替えたい場面がありました。

本来なら、手元で直してからデプロイの仕組みに乗せて転送すれば済む話です。実際そういう手順は用意してありました。なのにその時は、サーバーに入って対象ファイルを直接書き換えようとしました。1 行の JavaScript を直すだけだし、と。

これが全部の始まりでした。

書き込み先が権限の要る場所だったので、`sudo` を付けます。パスワードを対話で打つ代わりに、標準入力から渡そうとしました。だいたいこういう形です。

```bash
# やってしまった書き方（NG）
printf '%s\n%s' "$password" "$file_content" | sudo -S tee /var/www/.../app.js
```

`sudo -S` はパスワードを標準入力から読む、そう理解していました。だから 1 行目にパスワード、2 行目以降に中身を並べれば、パスワード行は sudo が食べて、残りが `tee` でファイルに書かれる。頭の中ではそうなっていました。

結果、そうはなりませんでした。

## 「1 行目は sudo が食べる」が崩れる瞬間がある

`sudo -S` の `-S` は「パスワードを **プロンプトするとき**、それを端末ではなく標準入力から読む」という意味です。ポイントは **プロンプトするとき** です。

sudo には認証キャッシュがあります。一度パスワードを通すと、しばらくの間は再入力を求めません。手元の環境だと 15 分くらいが既定です。この「しばらく」の間は、sudo はパスワードを聞きません。聞かないということは、**標準入力の 1 行目を読み取らない**ということです。

さっきのコマンドの直前に、私は別の `sudo` コマンドを 1 つ成功させていました。つまり認証がキャッシュされて温まっている状態でした。

その状態で先ほどのコマンドを流すと、こうなります。

- sudo はパスワードを聞かない
- だから標準入力の 1 行目（パスワード）を消費しない
- 消費されなかったパスワード行が、そのまま後ろの中身と一緒に `tee` へ渡る
- `tee` は渡された全部をファイルに書く
- 結果、**ファイルの 1 行目にパスワードが書かれる**

「1 行目は必ず sudo が食べてくれる」という暗黙の前提が、認証キャッシュ 1 つで崩れる。ここで 30 分溶かしました。最初は「なんで空になるんだ」「なんで先頭に変な行があるんだ」と、症状のほうばかり見ていて、原因にたどり着くのが遅れました。

![sudo -S の標準入力処理。キャッシュ有無で結果が反転する対比図](https://raw.githubusercontent.com/ishizakahiroshi/qiita-content/main/public/images/2026-07-07_sudo-s-stdin-password-leak_fig1.png)

上の図は、同じコマンドでもキャッシュの有無で結果が変わることを並べたものです。上段はパスワードを聞かれる（＝1 行目が消費される）ので中身だけが書かれます。下段はパスワードを聞かれないので、1 行目がそのまま中身に残ります。コマンドは 1 文字も違わないのに、直前に sudo を使ったかどうかだけで結果が反転します。

## なぜ「ただの漏洩」で済まなかったか

問題が二重だったのが、この事故の嫌なところでした。

1 つは、パスワードがファイルに書かれたこと。

もう 1 つは、そのファイルが **認証なしで配信される場所** に置かれていたことです。ブラウザで動くアプリだと、JavaScript や CSS はログインする前にブラウザがダウンロードします。だから配信サーバーは、そういう静的ファイルを認証なしで誰にでも返します。これは仕組みとして正しくて、脆弱性ではありません。守るべきなのは中のデータで、それはログインの先の API が守ります。

だから「静的ファイルが認証なしで見える」こと自体は普通です。問題は、**その普通に公開される場所へ、秘密を書き込んでしまった**ことのほうでした。公開されるのが正常だからこそ、そこに秘密が乗ると即座に外へ出ます。

幸い、書き込まれていた時間はごく短く、すぐ気づいて消しました。ただ「短かったから大丈夫」とは言えません。露出したパスワードは、後で変えるまで漏れた前提で扱うしかないです。

## 直し方は「秘密と中身を同じ入り口に混ぜない」

復旧のときに使った、そして最初からこうすべきだった書き方はこれです。

```bash
# 安全な書き方
tmp=$(mktemp)
printf '%s' "$file_content" > "$tmp"
sudo cp "$tmp" /var/www/.../app.js
rm -f "$tmp"
```

中身は一時ファイルに書いておいて、それを `sudo cp` で置くだけです。パスワードは標準入力に一切混ぜません。だから sudo の認証状態が中身に影響しようがありません。キャッシュが温まっていようがいまいが、ファイルには中身しか入りません。

![NG（混ぜる）と OK（一時ファイル + sudo cp）の経路対比図](https://raw.githubusercontent.com/ishizakahiroshi/qiita-content/main/public/images/2026-07-07_sudo-s-stdin-password-leak_fig2.png)

上の図の左が混ぜる書き方、右が分ける書き方です。左は認証の入り口（標準入力）とデータの入り口が同じなので、片方の前提が崩れるともう片方に漏れます。右は入り口が別なので、そもそも混ざりようがありません。

パスワードをどうしても標準入力から渡したいなら、せめて中身は別ファイルにします。「認証情報を運ぶ経路」と「データを運ぶ経路」を 1 本にまとめない。これが一番の教訓でした。

## 学んだこと

- `sudo -S` は「パスワードをプロンプトするときだけ」標準入力の 1 行目を読む。認証キャッシュ・`NOPASSWD`・sudo のバージョン差などで「聞かれない」と、1 行目は消費されず後ろに残る
- 「1 行目は必ず sudo が食べる」という前提でパイプを組むと、その前提が崩れた瞬間に秘密がデータ側へ漏れ落ちる
- 秘密（認証情報）とデータ（ファイルの中身）を同じ標準入力に混ぜない。中身は一時ファイルに書いて `sudo cp` で置く
- 公開される場所へ書き込む処理は、そもそも秘密が絶対に混ざらない経路で組む。「短時間だから」は言い訳にならない。露出したら変えるしかない
- そして一番は、用意された安全な手順を飛ばして本番を直接いじらないこと。今回の事故は、手元で直してデプロイに乗せればまるごと起きなかった

## 締め

コマンドの `-S` みたいな小さいオプション 1 つの挙動が、キャッシュの有無で反転する。頭では分かっていたつもりでも、いざ手を動かすと前提のほうが静かに崩れていました。

派手な脆弱性ではなくて、パイプの一行に秘密を混ぜたという、地味で防げたはずのミスです。だからこそ、次からは「認証とデータを同じ管に流さない」を、手が勝手にそう書くくらいまで習慣にしておきたいです。小さく。そこだけ、直します。
